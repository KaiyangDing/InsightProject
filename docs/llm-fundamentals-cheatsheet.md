# 应用岗大模型底层原理 · 速通清单

> 用法：每条都按 **「原理 → 对工程的影响 → 我在项目里怎么用到」** 三段式来讲，最显水平。
> 不用背数学，能用大白话讲清 + 知道它如何影响工程决策即可。

---

## 1. Transformer & Self-Attention

**一句话**：Transformer 用 self-attention 让序列里每个 token 都能"看到"并加权融合其他 token 的信息，从而建模上下文关系。

**2 分钟讲清**
- 核心是 attention：每个 token 生成 Query/Key/Value 三个向量；用 Q·K 算出对其他所有 token 的注意力权重，再对 V 加权求和，得到融合上下文的新表示。
- Multi-head：并行多组 attention，从不同子空间捕捉不同关系（语法、指代等）。
- 相比 RNN：可并行 + 能直接建模长距离依赖 —— 这是它能 scale 的根本。
- LLM 是 decoder-only + 因果掩码：生成时每个位置只能看左边，保证自回归。
- 复杂度 O(n²)：序列翻倍，attention 计算量约 4 倍 —— 长上下文贵的根源。

**高频面试题**：为什么 Transformer 取代了 RNN？self-attention 复杂度多少，对长文本意味着什么？
**强答要点**：并行化 + 长距离依赖；O(n²) 导致长上下文计算/显存激增（引出 KV cache、flash attention）。

---

## 2. Tokenization（分词）

**一句话**：模型不认字只认 token（子词单元），文本先被切成 token id 序列再输入。

**2 分钟讲清**
- 主流用 BPE/BBPE 子词算法：常见词一个 token，生僻词拆成多个。
- token ≠ 字 ≠ 词。英文 1 token ≈ 0.75 词；中文一个汉字常是 1–2 个 token，所以**中文更费 token**。
- 影响三件事：① 成本（按 token 计费）② 上下文窗口（按 token 算）③ 字符级任务弱（数字母、反转字符串），因为它看的是 token 不是字符。
- 工程含义：估算成本/裁剪上下文要按 token 算（tiktoken 等）；prompt 压缩能直接省钱。

**高频面试题**：为什么大模型经常数不对单词字母 / 算不准位数？
**强答要点**：处理的是 token 而非字符，字符边界信息被打包丢失；这类任务交给工具（代码执行），别让模型硬算。

---

## 3. 上下文窗口 & KV Cache（含 prompt caching）

**一句话**：上下文窗口是一次能"看到"的最大 token 数；KV cache 缓存历史 token 的 Key/Value 以避免生成时重复计算。

**2 分钟讲清**
- 自回归是一个 token 一个 token 出。每出一个新 token，理论上要重算前面所有 token 的 attention —— KV cache 把前面 token 的 K、V 存下复用，只算新 token，大幅加速。
- KV cache 占显存且随上下文长度线性增长 —— 长上下文部署贵、并发受限的主因之一。
- **Prompt caching**：把不变前缀（system prompt、长文档）的 KV 状态缓存复用，下次同前缀直接命中，跳过重复 prefill → 降本降延迟。懂 KV cache 才能解释它为何有效。
- **Lost in the middle**：上下文太长时，中间位置信息利用率下降 —— 关键信息放 prompt 首尾。

**高频面试题**：prompt caching 为什么省钱？为什么上下文越长越贵？
**强答要点**：贵在 prefill 要对全部输入算 O(n²) attention + KV cache 显存随长度增长；caching 复用固定前缀 KV、跳过重复 prefill，省的是计算与延迟（命中部分按更低价计费）。

## 4. 采样 / 解码参数

**一句话**：模型每步输出的是下一个 token 的概率分布；采样参数决定怎么从分布里挑 token，控制随机性。

**2 分钟讲清**
- temperature：缩放 logits。越高越随机/有创意，越低越确定；0 ≈ 贪心（greedy）每次选最高概率。
- top-p（nucleus）：只在累计概率达 p 的最小 token 集合里采样，动态截断长尾。
- top-k：只在概率最高的 k 个里采样。
- 工程含义：要稳定/可复现（SQL 生成、结构化输出、eval）→ temperature 调 0 或很低；要多样性（头脑风暴、写作）→ 调高。通常 temperature 与 top-p 调一个即可。
- 注意：即便 temperature=0 也不保证 100% 确定（受硬件/并行/MoE 路由影响）。

**高频面试题**：做结构化输出（JSON/SQL）时 temperature 怎么设，为什么？
**强答要点**：设 0 或接近 0 求确定性与可复现；配合 JSON mode / 约束解码 / schema 校验兜底。

## 5. Embedding & 向量检索（RAG 基础）

**一句话**：embedding 把文本映射成高维向量，语义相近的文本向量也相近；RAG 靠它做语义检索。

**2 分钟讲清**
- embedding 模型把文本编码成定长向量；用余弦相似度/内积衡量语义接近度。
- RAG 流程：文档 chunking → embedding → 存向量库 → query 也 embedding → 检索 top-k → 拼进 prompt 给 LLM 生成。
- 关键工程点：chunk 大小与重叠、embedding 选型、**rerank**（cross-encoder 精排）、**混合检索**（向量 + 关键词 BM25）、元数据过滤。
- 为什么要 RAG：补充知识截止外/私有/实时知识，可溯源，比微调便宜灵活。

**高频面试题**：RAG 检索效果不好，你从哪些方面优化？
**强答要点**：chunking 策略、更强 embedding、加 rerank、混合检索、query 改写/扩展(HyDE)、元数据过滤、调 top-k、建检索 eval 集量化召回/精确率。

## 6. 训练流程：Pretrain → SFT → RLHF/DPO

**一句话**：先海量文本自监督预训练学"语言和知识"，再用指令数据微调学"听话"，最后用偏好对齐学"符合人类偏好"。

**2 分钟讲清**
- Pretrain：next-token prediction，海量无标注语料，学语言能力与世界知识（最贵）。产物是只会续写的基座模型。
- SFT（监督微调）：用（指令,理想回答）教模型遵循指令、对话。
- RLHF / DPO（偏好对齐）：用人类偏好（A 比 B 好）让输出更有用/无害/诚实。RLHF 用奖励模型+PPO；DPO 跳过奖励模型直接优化，更简单稳定，现在很流行。
- 工程含义：知识来自 pretrain（所以有截止日期、改不动）；听话来自 SFT/对齐。改知识用 RAG，不指望对齐。

**高频面试题**：RLHF 和 DPO 区别？为什么有知识截止，怎么解决？
**强答要点**：RLHF 需训练奖励模型再 RL(PPO)，复杂；DPO 直接用偏好数据做对比损失，简单稳定。截止源于 pretrain 语料时间点；靠 RAG/联网工具解决，而非微调（微调适合改风格/格式/能力，不适合灌大量新事实）。

## 7. 幻觉：成因与缓解

**一句话**：幻觉是模型生成看似合理但实际错误/虚构的内容；根源是它本质在"按概率续写最像的答案"，没有事实校验机制。

**2 分钟讲清**
- 成因：① 训练目标是拟合语言分布而非求真 ② 知识覆盖不全时会"猜" ③ 长/模糊 prompt 加剧 ④ 对齐可能训得"过度自信不愿说不知道"。
- 缓解（应用层）：RAG 提供事实 + **强制引用溯源**；允许"无依据就答不知道"；结构化输出 + 校验；工具/代码执行替代心算；Critic/自我反思二次校验；降低 temperature；eval 专门测事实性。
- 这正是数据分析 agent 的核心设计点（"结论必须绑定真实查询结果"）。

**高频面试题**：你会怎么降低 agent 的幻觉？
**强答要点**：事实外置（RAG/工具）+ 强制溯源 + 结构化校验 + 反思/Critic + 低温度 + 事实性 eval；口号是"让模型推理、让工具提供事实"。

## 8. Function Calling / Tool Use 的真实原理

**一句话**：模型本身不执行任何工具，它只是按 schema 输出一个"该调哪个工具、参数是什么"的结构化请求，由你的代码去执行再把结果喂回去。

**2 分钟讲清**
- 你把工具名/描述/参数 schema 放进请求；模型决定要不要调、调哪个、填什么参数，输出结构化(JSON) 的 tool call。
- **执行是你的程序做的**，不是模型。结果作为一条消息回传，模型据此继续推理或给最终答复。
- 这就是 agent 循环本质：think → tool call → 你执行 → observe → 再 think …（ReAct）。
- 工程要点：工具描述要清晰（模型靠它选工具）、参数要校验、处理模型乱调/调错、用步数预算防死循环。

**高频面试题**：Agent 是怎么"调用"工具的？模型真能执行代码吗？
**强答要点**：模型只生成结构化调用意图，真正执行由外部代码/沙箱完成，结果回灌形成闭环；强调安全（沙箱、白名单、校验）与防循环。

## 9. 微调 vs RAG vs Prompt：怎么选

**一句话**：改"知识/事实"用 RAG，改"行为/风格/格式/特定能力"用微调，先用 prompt 能解决就别上重武器。

**2 分钟讲清**
- Prompt engineering：最快最便宜，先试。few-shot、CoT、结构化指令往往够用。
- RAG：需要私有/实时/大量事实、要可溯源、知识常更新 → 用 RAG。改知识不该微调。
- 微调（LoRA 等）：要稳定输出风格/格式、领域语气、压缩 prompt 降本、固化特定任务模式、或小模型替大模型降本 → 用微调。需高质量数据集。
- 常组合：微调一个听话小模型 + RAG 提供知识。
- 口诀：能 prompt 不 RAG，能 RAG 不微调，除非有明确"行为/成本"理由。

**高频面试题**：什么时候该微调，什么时候用 RAG？
**强答要点**：知识/事实→RAG（便宜、可更新、可溯源）；行为/风格/格式/能力固化或降本→微调；二者不互斥常配合；先穷尽 prompt。

## 10. MoE & Scaling Law（概念，能聊即可）

**一句话**：Scaling law 说性能随参数/数据/算力按幂律提升；MoE 用"稀疏激活"在不成比例增加计算的前提下做大参数量。

**2 分钟讲清**
- Scaling law：能力随规模（参数、数据、计算）可预测提升 —— "大力出奇迹"的理论依据，指导算力/数据配比（如 Chinchilla）。
- MoE（混合专家）：把 FFN 拆成很多"专家"，每个 token 只路由到少数专家激活 → 总参数大但单次推理只算一小部分，性价比高。DeepSeek-V3、部分 Qwen 都是 MoE。
- 工程含义：MoE 推理便宜但部署要装下全部专家（显存有特点）；能解释"为什么 DeepSeek 又大又便宜"。

**高频面试题**：DeepSeek 为什么又强又便宜？MoE 是什么？
**强答要点**：MoE 稀疏激活（大总参数、小激活计算）+ 训练/推理工程优化（MLA、FP8、prompt caching 等）；Scaling law 指导规模与数据配比。

---

## 11. 上下文工程（Context Engineering）

**一句话**：在有限上下文窗口里，"放什么、怎么组织、何时压缩"比单纯堆 prompt 更决定 agent 表现。

**2 分钟讲清**
- 长任务/多轮/多工具会让上下文爆炸，必须主动管理：摘要历史、结构化 scratchpad、只检索相关片段、长结果落盘只留引用。
- 关键信息放首尾（对抗 lost-in-the-middle）。
- 区分短期记忆（本次任务 scratchpad）与长期记忆（跨会话知识，存向量库/DB）。
- agent 工程当下最热能力之一，也是本项目长链路分析的核心。

**高频面试题**：agent 跑长任务上下文超了怎么办？
**强答要点**：摘要 + 检索 + 外置存储 + 结构化状态 + 关键信息定位；强调"上下文是预算，要主动管理而非被动塞满"。
